# Optional Feature: CalDAV endpoint (read-write) via API keys

> Status: `[ ]` planned — not started

## Problem

The existing `/ical?secret=<uuid>` route is read-only and uses a separate credential from the API key system. CalDAV would let users subscribe to Auto-Cal from native calendar apps (Apple Calendar, Thunderbird, etc.) using an API key, and allow those apps to push back completions and deletions.

---

## Write semantics

| CalDAV operation | Auto-Cal action |
|---|---|
| `PUT` with `STATUS:COMPLETED` | `myCompleteTodo` / `myCompleteHabit` |
| `DELETE /<uid>.ics` | `myDeleteTodo`; habit instance → no-op (re-appears on next writeback) |
| `PUT` with new UID | **403** — creation goes through the app UI only |
| `PUT` with changed `DTSTART`/`DTEND` | **403** — scheduling is auto-managed |

---

## Route surface

Mount: `app.use('/caldav', createCaldavRouter(db))`

```
OPTIONS  /caldav/:userId/                    200 + Allow + DAV headers
PROPFIND /caldav/:userId/                    207 (principal + calendar-home-set)
PROPFIND /caldav/:userId/events/             207 (collection — all event hrefs)
PROPFIND /caldav/:userId/events/:uid.ics     207 (single event: ETag, content-type)
GET      /caldav/:userId/events/:uid.ics     200 text/calendar (single VEVENT)
HEAD     /caldav/:userId/events/:uid.ics     200 headers only
PUT      /caldav/:userId/events/:uid.ics     204 (update event state)
DELETE   /caldav/:userId/events/:uid.ics     204 (delete event)
REPORT   /caldav/:userId/events/             207 with matching events
*        anything else                       405 Method Not Allowed
```

---

## UID scheme

Format: `todo-<id>-<weekStart>@auto-cal` or `habit-<id>-<instanceIndex>-<weekStart>@auto-cal`

URL segment is the UID without `@auto-cal`:
`/caldav/:userId/events/todo-<id>-2026-01-05.ics`

```typescript
function parseUid(uid: string): { kind: 'todo' | 'habit'; id: string; weekStart: string; instanceIndex?: number }
```

---

## Auth (`caldav-auth.ts`)

Accepts three credential forms (in precedence order):

1. `Authorization: Bearer acal_xxx` — same pipeline as `/graphql`
2. `Authorization: Basic <base64(any:acal_xxx)>` — password treated as API key
3. `Authorization: Basic <base64(any:<icalSecret-uuid>)>` — backward compat with existing secret

Scope enforcement: `read` scope for GET/PROPFIND/REPORT; `write` scope for PUT/DELETE.

## Key decisions / tradeoffs

- **No `caldav-adapter`** — the only maintained library is Koa-based; hand-rolling the surface is less work than adapting it to Express.
- **Hand-rolled XML** — responses are short fixed-structure documents; template strings suffice. Add `xmlbuilder2` if more complex XML is needed later.
- **No `/.well-known/caldav` redirect in v1** — users enter the full CalDAV URL from the settings page. Add the redirect as a follow-up.
- **`Depth: infinity` → 403** with `<d:propfind-finite-depth/>` error body.
- **ETag** = `SHA-256(event-ics-string).slice(0, 16)`, quoted. Supports `If-None-Match` (GET → 304) and `If-Match` validation (PUT/DELETE → 412 on mismatch).
- **Habit DELETE** — no-op: the instance is virtual, re-generated by the scheduler on the next writeback. Skipping is a separate feature (#14).
